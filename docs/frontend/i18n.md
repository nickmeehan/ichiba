# Internationalization

Nimbus supports multiple languages and locales using react-i18next. Translation files are managed through a dedicated workflow with Crowdin, and the app detects the user's preferred locale automatically. Currently supported locales are English (en-US), Spanish (es-ES), French (fr-FR), German (de-DE), Japanese (ja-JP), and Portuguese (pt-BR).

## react-i18next Setup

The i18n instance is initialized in `src/i18n/index.ts` with the following configuration:

- **Backend**: Translations are loaded lazily per namespace using `i18next-http-backend`, fetched from `/locales/{lng}/{ns}.json`.
- **Detection**: Locale is detected in order: user profile setting, browser `navigator.language`, `Accept-Language` header, fallback to `en-US`.
- **Namespaces**: Translations are split by domain: `common`, `dashboard`, `projects`, `tasks`, `admin`, `settings`, `errors`. Each page loads only the namespaces it needs.
- **Fallback**: Missing keys fall back to `en-US`. In development, missing keys log a warning to the console.

```ts
i18n.use(HttpBackend).use(LanguageDetector).use(initReactI18next).init({
  fallbackLng: "en-US",
  ns: ["common"],
  defaultNS: "common",
  interpolation: { escapeValue: false },
});
```

## Translation Workflow

1. Developers add English strings using the `t()` function with descriptive keys: `t("tasks.createModal.title")`.
2. On merge to `main`, a GitHub Action uploads new keys to Crowdin.
3. Translators work in Crowdin and submit translations for review.
4. Approved translations are pulled back via a weekly automated PR.
5. The CI pipeline checks for missing translations and warns if any supported locale is below 95% coverage.

### Key Naming Conventions

- Use dot-separated namespaced keys: `namespace.section.element`
- Use descriptive English-like keys, not numeric IDs
- Plurals use `_one`, `_other` suffixes: `tasks.count_one`, `tasks.count_other`
- Context variants use `_male`, `_female` where grammatically needed

## Date and Number Formatting

Never format dates or numbers manually. Use the `useLocaleFormat` hook which wraps the `Intl` API:

```tsx
const { formatDate, formatNumber, formatCurrency } = useLocaleFormat();

formatDate(task.dueDate, "medium");      // "Mar 15, 2026" or "15 mar 2026"
formatNumber(1234.5);                     // "1,234.5" or "1.234,5"
formatCurrency(99.99, "USD");            // "$99.99" or "99,99 $"
```

Relative dates (e.g., "2 hours ago") use `Intl.RelativeTimeFormat` via the `useRelativeTime` hook.

## RTL Support

Nimbus supports right-to-left languages (Arabic and Hebrew are on the roadmap). The layout system uses CSS logical properties (`margin-inline-start` instead of `margin-left`, `padding-inline-end` instead of `padding-right`).

The `dir` attribute is set on `<html>` based on the active locale. All Flexbox layouts use `gap` and logical alignment properties to flip automatically. Icons that imply direction (arrows, chevrons) are mirrored via `transform: scaleX(-1)` in RTL mode.
