# Reporting

Nimbus provides analytics dashboards and scheduled reports powered by ClickHouse. The reporting system is designed for fast analytical queries over large datasets while maintaining clear data freshness guarantees for end users.

## ClickHouse Query Patterns

ClickHouse is the analytics database for all reporting queries. Data is synced from PostgreSQL via ETL pipelines (see [ETL Pipelines](etl-pipelines.md)).

### Common Query Patterns

**Task velocity by sprint:**
```sql
SELECT
  sprint_id,
  countIf(status = 'completed') AS completed,
  countIf(status = 'in_progress') AS in_progress,
  count() AS total
FROM analytics.tasks
WHERE tenant_id = {tenantId:UUID}
  AND sprint_id IN ({sprintIds:Array(UUID)})
GROUP BY sprint_id
```

**Cycle time distribution (time from created to completed):**
```sql
SELECT
  quantile(0.5)(completed_at - created_at) AS median_cycle_time,
  quantile(0.9)(completed_at - created_at) AS p90_cycle_time,
  quantile(0.99)(completed_at - created_at) AS p99_cycle_time
FROM analytics.tasks
WHERE tenant_id = {tenantId:UUID}
  AND completed_at IS NOT NULL
  AND created_at >= {startDate:Date}
```

All queries use parameterized placeholders. Never interpolate user input into query strings.

## Materialized Views

ClickHouse materialized views pre-aggregate data for dashboard widgets that would otherwise require expensive scans:

| View | Source | Aggregation | Refresh |
|------|--------|-------------|---------|
| `mv_daily_task_counts` | `analytics.tasks` | Daily count by status per project | On insert |
| `mv_weekly_velocity` | `analytics.tasks` | Weekly completed task count per team | On insert |
| `mv_user_activity_hourly` | `analytics.events` | Hourly event count per user | On insert |
| `mv_project_health` | `analytics.tasks` | Overdue ratio, avg cycle time per project | On insert |

Materialized views use ClickHouse's `AggregatingMergeTree` engine for incremental aggregation. They update automatically as new data arrives from ETL pipelines.

## Report Scheduling

Tenants can schedule recurring reports that are generated and delivered via email or Slack:

| Report | Available Frequencies | Format | Delivery |
|--------|----------------------|--------|----------|
| Sprint Summary | Weekly, biweekly | PDF, CSV | Email, Slack |
| Team Velocity | Weekly, monthly | PDF | Email |
| Task Aging | Daily, weekly | CSV | Email, Slack |
| Resource Utilization | Weekly, monthly | PDF | Email |
| Custom Query | Any cron expression | CSV | Email, S3 |

Reports are generated by a Temporal workflow that:
1. Runs the report query against ClickHouse
2. Renders the results into the requested format (PDF via Puppeteer, CSV via streaming writer)
3. Uploads to S3 and sends the delivery notification with a time-limited download link

## Data Freshness Guarantees

Each dashboard widget displays a "Data as of" timestamp showing when the underlying data was last updated. Freshness targets:

| Data Type | Max Staleness | Mechanism |
|-----------|--------------|-----------|
| Task counts | 5 minutes | CDC pipeline runs every 5 minutes |
| Event activity | 1 minute | Event pipeline runs every minute |
| Aggregated metrics | 15 minutes | Materialized view refresh + ETL |
| Scheduled reports | Reflects last full ETL run | Report generation waits for pipeline completion |

If data is staler than the target, the widget shows a warning icon. Users can click to see the exact pipeline status.

## Tenant Isolation

All reporting queries are scoped by `tenant_id`. ClickHouse row-level security is enforced via a query rewriting middleware that injects `AND tenant_id = {currentTenantId}` into every query. This is defense-in-depth on top of the application-level tenant scoping.
