# CI Test Configuration

This guide covers how tests are executed in CI, including parallel execution, test splitting, flaky test management, and result reporting.

## Parallel Test Execution

Tests run in parallel across multiple CI runners to minimize pipeline duration:

| Test type | Parallelism | Strategy |
|-----------|------------|----------|
| Unit tests | 4 shards | Jest `--shard` flag |
| Integration tests | 2 shards | Jest `--shard` flag |
| E2E tests | 4 workers | Playwright `--workers` |
| Contract tests | 1 runner | Sequential (fast enough) |

```yaml
# .github/workflows/ci.yml (unit test job)
strategy:
  matrix:
    shard: [1, 2, 3, 4]
steps:
  - run: pnpm test:unit --shard=${{ matrix.shard }}/4
```

## Test Splitting

Jest shards split tests by file. For balanced splitting, the CI pipeline uses test timing data from previous runs:

1. After each CI run, test durations are uploaded to S3 (`nimbus-ci-artifacts/test-timings/`).
2. On the next run, the timing data is downloaded and used to balance shards.
3. If no timing data is available (new test files), files are split alphabetically.

This keeps shard execution times within 10% of each other.

## Flaky Test Detection

A test is considered flaky if it passes on retry after an initial failure. The CI pipeline tracks flaky tests:

1. All test jobs use `--retries 2` for E2E and `--bail 0` for Jest.
2. The `flaky-test-reporter` action analyzes test results and identifies tests that failed then passed.
3. Flaky tests are reported to the `#nimbus-dev` Slack channel weekly.
4. A test that is flaky 3+ times in a month gets a `flaky-test` Linear ticket auto-created.

### Quarantine Process

Persistently flaky tests (failing > 10% of runs) are quarantined:

```typescript
// Mark a test as quarantined (skipped in CI, tracked for fixing)
it.skip('flaky: renders dashboard after slow API response', () => {
  // TODO: Fix flakiness - NIMB-4567
});
```

Quarantined tests are tracked in the `#nimbus-test-health` dashboard.

## Test Reporting

Test results are reported through multiple channels:

| Report | Destination | Contents |
|--------|------------|----------|
| JUnit XML | GitHub Actions | Test results shown in PR checks |
| Coverage report | Codecov | Coverage diff on PR |
| Chromatic report | Chromatic | Visual regression results |
| k6 results | Datadog | Load test metrics (nightly only) |

JUnit reports are generated by adding `--reporters=jest-junit` and uploaded as CI artifacts.

## Failure Notifications

| Failure type | Notification |
|-------------|-------------|
| PR test failure | GitHub PR check (blocking) |
| `main` branch failure | `#nimbus-dev` Slack alert |
| Nightly test failure | `#nimbus-dev` Slack + PagerDuty (P2) |
| Coverage drop > 2% | Codecov PR comment (blocking) |

The `main` branch must always be green. A failing `main` build is treated as a P2 incident.
