# Audit Log Service

The audit log service maintains an immutable record of all significant actions performed within a Nimbus workspace. It serves both compliance requirements and operational troubleshooting needs.

## Event Capture

Audit entries are generated by a middleware-style event handler that subscribes to domain events. When an auditable event occurs, the handler records:

| Field         | Description                                    | Example                        |
|---------------|------------------------------------------------|--------------------------------|
| `event_type`  | The domain event name                          | `task.updated`                 |
| `actor_id`    | The user who performed the action              | `usr_abc123`                   |
| `tenant_id`   | The workspace where the action occurred        | `ws_def456`                    |
| `resource_type`| The entity type affected                      | `task`                         |
| `resource_id` | The specific entity ID                         | `tsk_ghi789`                   |
| `changes`     | A JSON diff of the before/after state          | `{"status": ["open", "done"]}` |
| `ip_address`  | The client IP address                          | `203.0.113.42`                 |
| `user_agent`  | The client user agent string                   | `Mozilla/5.0 ...`             |
| `timestamp`   | When the action occurred (server time, UTC)    | `2026-03-15T14:30:00Z`        |

## Storage

Audit log entries are stored in a dedicated `audit_logs` table in PostgreSQL. This table is append-only; no UPDATE or DELETE operations are permitted. A database trigger rejects any attempt to modify existing rows.

For high-volume workspaces, older audit entries are automatically archived to S3 in Parquet format after 90 days, with the PostgreSQL rows retained for 1 year before deletion.

## Structured Log Format

Each audit entry is also emitted as a structured JSON log line for ingestion by the centralized logging system. This allows correlation between audit events and application logs using the shared `requestId`.

```json
{
  "level": "info",
  "service": "audit",
  "event_type": "task.updated",
  "actor_id": "usr_abc123",
  "tenant_id": "ws_def456",
  "resource": "task:tsk_ghi789",
  "requestId": "req_xyz"
}
```

## Retention Policies

| Tier       | Active Retention | Archive Retention | Export   |
|------------|-----------------|-------------------|----------|
| Free       | 30 days         | None              | No       |
| Pro        | 1 year          | 2 years (S3)     | CSV      |
| Enterprise | 2 years         | 7 years (S3)     | CSV, JSON|

## Compliance Queries

The audit log API supports filtering by actor, resource type, date range, and event type. This enables compliance teams to answer questions like "Who modified this task and when?" or "List all admin permission changes in the last quarter."

## Export

Enterprise workspaces can export audit logs in CSV or JSON format. Exports run as background jobs and are delivered as signed download links with a 24-hour expiration.
