# Debugging Guide

This guide covers the tools and techniques for debugging issues across the Nimbus stack, from frontend React components to backend services and database queries.

## Structured Logging

All Nimbus services use [Pino](https://getpino.io/) for structured JSON logging. Every log entry includes a `correlationId` that ties together all log lines for a single request.

```bash
# Tail API logs locally
docker compose logs -f api | pnpm pino-pretty

# Filter by correlation ID
docker compose logs api | pnpm pino-pretty | grep "abc-123-def"
```

In production, logs are shipped to Datadog. Use the Logs Explorer with the query:
```
service:nimbus-api @correlationId:"abc-123-def"
```

## Request Tracing with Correlation IDs

Every inbound HTTP request receives a `X-Correlation-ID` header (generated by the API gateway if not already present). This ID propagates through:

1. API request context (`req.correlationId`)
2. Background jobs (stored in job metadata)
3. Database query comments (via Prisma middleware)
4. Downstream service calls (forwarded in headers)

To find all activity for a user report, grab the correlation ID from the response headers or from the Datadog APM trace.

## Database Query Debugging

Enable Prisma query logging in development:

```bash
# In .env.local
DATABASE_LOG_LEVEL=query
```

For slow query analysis, connect to the local PostgreSQL instance:

```bash
docker compose exec postgres psql -U nimbus -d nimbus_dev
```

```sql
-- Find slow queries
SELECT query, mean_exec_time, calls
FROM pg_stat_statements
ORDER BY mean_exec_time DESC
LIMIT 20;
```

## Frontend Debugging (React)

- **React DevTools**: Install the browser extension. Use the Profiler tab to identify unnecessary re-renders.
- **React Query DevTools**: Enabled by default in development. Shows cache state, query timing, and refetch behavior.
- **Network tab**: Filter by `/api/` to see all Nimbus API calls. Check response payloads and timing.

For state-related bugs, add `window.__NIMBUS_DEBUG__ = true` in the browser console to enable verbose React Query logging.

## Production Debugging

Production debugging requires additional caution:

1. **Never** connect directly to a production database. Use the read replica via the `nimbus-prod-readonly` IAM role.
2. Use Datadog APM to trace slow requests. Look at the flame graph for time spent in each span.
3. For intermittent issues, create a Datadog monitor with the relevant query and wait for it to trigger.
4. If you need to add temporary logging, use a feature flag (`debug.verbose-logging`) scoped to the affected tenant.
